HTML5 Form Validation

Validating form data is a crucial task a developer needs to think through before getting the form out there, where teaming bots and hackers roam the web, looking for an opportunity to compromise server and innocent user's security. It may not be a permanet fix but it makes the life of the bad guys a little harder.

Phases of Form Validation

There are three stages a form can be validated before its contents are processed by the server:

  • HTML validation
  • JavaScript validation
  • Server-side validation

The first two techniques are commonly called front-end validation, because the validation is done at the user's browser before data is transmitted to the server. The third type of validation is also known as back-end validation, and takes place on the host server computer after the form has been submitted.

Front-end Vs Back-end Validation

Form validation is not automatic, the developer has to take deliberate steps to ensure form data is safe to be handled by the server. And, it starts at the front-end. That doesn't mean that back-end validation is less important, on the contrary, it is the most important. If the first two methods fail, the server-side validation is the last resort.

However, relying solely on the server to validate user input can overwork the server and slow other critcal processes, especially in a busy environment. This is because, when server validation detects form errors, the data has to be ralayed back to the user for correction.

The user then will have an opportunity to correct the errors and resend the form. This process will repeat as long as the form has invalid data. Too much of this back and forth on a single task creates an extra load on the server and can impact server performance, for a heavily used form. For this reason, form validation should start at the browser level and scale up to the server.

In this tutorial we will use the features provided by HTML5 to carry out form validation at the browser level. Let's begin with creating a form with five input fields: username, email, password, telephone and textarea. You can just copy the code below into your code editor and save the file as registration.html or any other name of your choice with the extension .html

<! DOCTYPE html>
<html lang="en">
<head>
	<meta charset="UTF-8">
	<title>Registration Form</title>
</head>
<body>
	<form method="POST" action="">
		<label for="username">Username:</label>
		<input type="text" name="username" maxlength="15" placeholder="Username ..." required>
		<label for="email">Email:</label>
		<input type="email" name="email" maxlength="20" placeholder="Email..." required>
		<label for ="tel">Telephone</label>
		<input type="tel" name="tel" maxlength="9"placeholder="Phone number..." required>
		<label for="password">Password</label>
		<input type="password" name="password" placeholder="Password..." required>
		<label for="comment">Comment</label>
		<textarea name="comment" col=30 rows=5 placeholder="Type your text here..." required></textarea>
		<input type="submit" name="submit" value="Submit">
	</form>
</body>
</html>

Before explaining the attributes used in the input fields above, let's style the form with some CSS to make it look a little bit decent. Copy the following styles into the <head> section of the registration.html document and save the changes.

HTML Form Attributes

Now let's explain the attributes used in the form.

The placeholder Attribute

The placeholder=" " guides the user where to type the values. For example placeholder="Username" tells the user where username should be typed. When the user starts typing in the field the placeholder text clears up. That is one of the differences between placeholder attribute and value=" " attribute, which developers sometime back used with text inputs to mimic a placeholder in older browsers.

The value Attribute

The value=" " attribute labels the input. For example a submit input button must have a value attribute, which is text showing the purpose of the button. It can take values such value="Submit" or value="Send".

If value attribute is used with text input fields, the user has to delete the text placed in the field and type new text, otherwise, the default text will be submitted with the form. To avoid such inconveniences the placeholder attribute was invented. Older browsers prior HTML5, however, do not support the placeholder attribute and will not display it. To make the form usable in those browsers, naming fields with the label attribute is necessary. For browsers that support HTML5, placeholders can be used in place of labels.

The maxlength Attribute

A maxlength=" " attribute is used to limit the number of characters that can be typed in a field. For example maxlength="10" will allow only ten characters in the field. This way the user cannot exceed the number of characters expected by the form owner.

Two things to take note of, though: One, maxlength does not give feedback to the user on the number of characters expected. Two, if the length is not chosen carefully, it may restrict users unnecessarily and upset them. In a business setting that can impact the business negatively.

The type Attribute

In HTML5, the type="" attribute has been expanded to include several other values that signify specific data items. For example type="email" will only accept valid email addresses. In the case of type="telephone" , only numbers and the '+' symbol for country code will be accepted, just like the type="password" will hide the real characters being typed. The type="url" attribute will only accept valid web addresses. So, you can see that using this attribute, you let the browser validate emails, urls or telephone numbers.

The required Attribute

If you apply the required attribute on any input in the form, that field will become mandatory, and cannot be left blank. If you try to submit the form while one of the required fields is not filled, the browser will refer you back to the field. Different browsers have their own way of highlighting required fields, if left blank.

You can deliberately mess around with these attributes and see how your form responds, before we turn to pattern attribute.

The pattern Attribute

Let's now turn to the pattern atttibute. This attribute was introduced in HTML5 to allow complex HTML validation techniques, using what is called regular expressions. Regular expressions are special defined pattern of characters that must be matched against what is typed in the field. The pattern attribute can be used with text , tel , date , email , password , url and search input types.

HTML5 Form Validation Using Regular Expressions

We are going to update our earlier form with the pattern attribute in each input we have used. The pattern attribute will enable us to evaluate the input value using a regular expression for a match. So, in essence you define a rule for the type of characters that should be allowed in the input, without which the value in the field will be deemed invalid. I'm intending to post a tutorial on regular expressions for those unfamiliar with these type of rules. Watch out for Regular Expressions in JavaScript in the upcoming post on this site.

As for now, we are going to use regular expressions that suit our purpose and we will also explain what they mean. Further, a global title attribute will used along with the pattern attribute to enable a pop up guide to the user.

Pattern and Title Attributes

<form method="POST" action="">
	<label for="username">Username:</label>
	<input type="text" placeholder="Username ..." pattern="\w+" title="Type only letters, numbers and underscore with no space" required>
	<input type="submit" name="submit" value="Submit">
</form>

The pattern above allows us to type one or more alphanumeric characters (a-zA-Z0-9) or underscore in the field. The \w stands for both lowercase and uppercase alphabet letters, numerals 0 - 9 and the underscore '_'. The backslash \ is telling the browser that "w" is a special characther and should not be treated literally. Also note that the w used here is lowercase. Uppercase W negates (a-zA-Z0-9).

The usage of + makes the expression expect at least one of the characters represented by \w. The expression \w can also be expressed as pattern="[a-z0-9_]+" and it would mean the same. Further, the number of characters allowed in the field can be restricted by using braces { } . For example \w{3,} would allow three or more characters. The pattern \w{3,7} restricts between three and seven characters and \w{,7} allows at most seven characters in the field.

Try the form with different character patterns and see how it responds at submission.

Validating Telephone Field

Let's try to validate a telephone field. Telephone number formats vary from country to country, but generally we expect a telephone number to have a country code that begins with a + symbol. If for instance, the country code is a three-digit number, we can represent that by starting the expression with \+[0-9]{3} . This expression uses a character class defined by [ ] for the digit range 0-9. The expression mean that the field values must start with + , then followed by exactly any three digits between 0 and 9.

We could achieve exactly the same effect by using a named character class pattern \+\d{3} . The \d stands for digits 0 - 9.

Next, we would like a hyphen - to connect the country code and the actual telephone number consisting of nine digits. Therefore, we can contruct a complete regular expression to evaluate such a number as follows:

<form method="POST" action="">
	<label for ="tel">Telephone</label>
	<input type="text" name="tel" placeholder="Phone number..." pattern="\+[0-9]{3}-[0-9]{9}" title="Tel format: +XXX-XXXXXXXXX" required>
	<input type="submit" name="submit" value="Submit">
	</form>

Notice that we have used type="text" attribute instead of tel just to illustrate how an expression can change a text field to accept only numbers and the + symbol.

In instances where there is no restriction regarding the number of digits to be used and how they should be grouped in a field, you could just use type="tel" and it would suffice.

When dealing with telephone numbers for a global audience, caution should be exercised as some countries use different telephone number formats. The type="tel" attribute would be ideal for such instances.

Validating Email Address

Suppose we wanted to validate an email address using a regular expression instead of the type="email" . In such cases care must be taken for the mandatory special characters that make an email valid. In addition, those special characters must also appear in their correct positions.

For example a valid email must have the @ symbol and it must not begin the email address. Further, there must be a dot . and it should not be ending the email address. After the last dot . there must be at least two letters. That is the pattern of most email addresses. And by convention, we use lowercase characters for email addresses.

Taking those considerations into account, we can build a regular expression like the one below.

<form method="POST" action="">
	E-mail: <input type="text" name="email" pattern="[a-z0-9._%+-]+@[a-z0-9.-]+.[a-z]{2,}$" title="Type a valid email address.">
</form>

I have changed the input type from email to text to be able to test the validation properly.

If you picked something from the immediate example before the one above, you will notice what this pattern is trying to achieve.

The pattern above means that the email address can begin with one or more alphanumeric characters (a - z) or (0 - 9), one or more . _ % + - characters, followed by a mandatory @ . What follows after the @ should be one or more alphanumeric characters including a dot . or hyphen - . Finally, there should be a mandatory dot '.' that is followed by at least two alphabet characters.

The $ symbol is used to mark the end of the matching pattern. Here, it means that the last two or more alphabet characters after the last dot should end the email address. Any other non-alphabet character after that would cause the email address to be invalid. The email address in this case must be typed in lowercase letters.

A pattern like this can be designed to filter emails of certain characteristic where the general type="email" attribute would not be sufficient.

Validating Passwords

This time we want to build an expression to validate passwords. Many establishements that collect user personal information devise password policies that try to make it harder to be cracked. Policies such as forcing users to use passwords that mix uppercase and lowercase letters, numbers and symbols. This trend truly results in stronger passwords, but the practice has proved to be counter productive. Instead, many organisations nowadays prefer using strong encrypting algorithms at the back-end where passwords are at a higher risk of being stolen than at the front-end.

In that case, it may not be much beneficial to restict users so much to the point where they may feel being panalised for using the web form. Nonetheless, moderate requirements is necessary. Let's consider the pattern below:

<form method="POST" action="">
	Password: <input type="password" name="pw" pattern="(?=.*d)(?=.*[a-z])(?=.*[A-Z]).{6,}" title="Must have at least one number and one uppercase and lowercase letter, and at least 6">
</form>

The pattern above restricts the password to be of at least one digit, at least one lowercase letter and at least one uppercase letter. In addition, the total number of characters of the password must be at least six. This time we have used regular expression special characters ( ) and ?= .

The ( ) is called capturing patenthesis. It matches what is inside the parenthesis and remembers the match. The ?= is called positive look ahead expession and matches what is listed on its left, if it is followed by what is listed on its right. The .* are two special characters where the dot . in this case stands for any single character except new line and the * matches 0 or more times the immediate character on its left.

Conclusion
